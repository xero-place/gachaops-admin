# ライブ映像切替機能 (Live Control)

実機ガチャマシンの再生映像を遠隔から即時に変更するための機能群です。
キャンペーン開始の合図、突発の不具合表示差し替え、夜間モード強制など、
時間割では拾えない運用上の即応性を提供します。

## 構成

3 つのエントリポイント、1 つの共通コンポーネント、1 つのクライアント状態ストアで構成されます。

### エントリポイント

| 場所 | 用途 | スコープ |
|---|---|---|
| `/live-control` | ライブ操作センター (新画面) | 全マシン / 店舗 / グループ / 個別 |
| `/devices` (一覧) | 行チェックボックス + フッターバー | 複数台 |
| `/devices/[id]` (詳細) | ヘッダーの「映像を切替」ボタン | 単台 |

### 共通 UI: `LiveControlSheet`

`src/components/domain/live-control-sheet.tsx`

呼び出し側はスコープ (端末ID 配列 + 表示用ラベル) を渡すだけで、以下を共通化:

1. **プログラム選択** (公開済プログラムのみ、検索付き)
2. **モード選択**:
   - **即時切替**: 端末を `manual` モードへ。手動で計画モードへ戻すまで継続
   - **期限指定**: 指定分後 (15/30/60/120 分プリセット or 任意) に自動で計画モード復帰
3. **確認** (対象台数 5 台超は警告表示)

### モード遷移

```
[plan モード] ──映像切替──→ [manual モード]
                  ↑                  │
                  │                  │
                  └─ 計画モードに戻す/期限切れ
```

`Device.play_mode: 'plan' | 'manual' | 'idle'` の遷移として表現。手動切替中の端末は UI 各所で ⚡ アイコン + 「手動切替中」バッジで識別されます。

### スコープ別の操作

#### 1台 (`/devices/[id]`)
- 端末詳細ヘッダーの「映像を切替」ボタン → シート起動
- 「現在の再生」セクションに切替元時刻 + 期限を表示
- 手動モード時は「計画モードに戻す」ボタンが現れる

#### 複数台 (`/devices`)
- 各行の左端にチェックボックス (オフライン端末は無効化)
- ヘッダーチェックボックスで「絞込中の全オンライン端末」をトグル
- フィルタバーの「絞込中の全端末を選択」ボタン
- 選択中はフッターに浮動バー: 「選択端末で映像を切替」 + 「計画モードに戻す」

#### 店舗単位 / グループ単位 / 全体 (`/live-control`)
- **クイック操作**: 全マシン一斉切替・全手動切替を解除
- **対象を選んで切替**: 店舗カード / グループカードをクリックでシート起動
- **再生中の映像**: フリート全体で同じ映像を再生中の端末数を集計
- **手動切替中の端末**: 期限とともに一覧、ボタン1つで個別解除
- **操作履歴**: 過去 50 件の切替・解除を時系列表示

## 状態管理 (`useLiveStore`)

`src/stores/live-control-store.ts` (Zustand)

モック用のクライアントサイド状態。本番接続後は **削除** され、サーバ状態 + WebSocket push に置き換わります。

```typescript
{
  overrides: Record<DeviceId, LiveOverride>; // 上書き状態
  history: LiveActionLog[]; // 操作履歴
  applySwitch: (...) => void;
  restorePlan: (...) => void;
}
```

`applyOverridesToDevices(devices, overrides)` で fixture の `play_mode`/`current_program_*` を上書きしたコピーを返し、UI はそれを表示する仕組み。

期限切れのオーバーライドは `applyOverridesToDevice` 内で自動フィルタされるため、`expires_at` を過ぎれば各画面は自然に元の状態に戻ります (UI 側のみ; Mock API のフィクスチャ自体は不変)。

## API マッピング (Step B / OpenAPI v0.1)

このフィーチャは既存のエンドポイントで実装可能で、新規エンドポイントは不要です。

| UI 操作 | エンドポイント | リクエスト |
|---|---|---|
| 単台切替 | `POST /devices/{id}/commands` | `{ type: "play_program", program_id, expires_at? }` |
| 複数台/グループ/全体切替 | `POST /devices/bulk` | `{ device_ids?: [], group_id?, all?: true, command: { ... } }` |
| 計画モード復帰 (単台) | `POST /devices/{id}/commands` | `{ type: "play_program", program_id: null }` (= plan モード再開) |
| 計画モード復帰 (一括) | `POST /devices/bulk` | 同上 |
| 操作履歴の参照 | `GET /audit-logs?action=command` | filter on `command` action type |

実装時は `commands` の `type: "play_program"` ペイロードに `expires_at` を追加することで「期限指定」モードをサポートします (現 OpenAPI v0.1 の `CommandSimple` を `CommandPlayProgram` に拡張)。

## WebSocket 連携 (Step D)

実機への配送は WebSocket で行います:

```json
{
  "type": "play_program",
  "message_id": "m_xxx",
  "payload": {
    "program_id": "prg_1",
    "expires_at": "2026-05-10T11:00:00Z"
  }
}
```

`/ws-console` の既存テンプレ `play_program` がこの形式の動作確認に使えます。

連動再生グループ (`device_group.linked = true`) では、サーバ側がタイムスタンプを揃えて全メンバー端末へ同時送出することで同期再生を実現します (具体プロトコルは `Step D`)。

## オフライン端末の扱い

- UI: 一覧のチェックボックスは無効化、詳細の「映像を切替」ボタンも `disabled`
- ライブ操作センターでスコープ選択時は、オンライン端末のみカウント対象
- もし一括送信時に対象に含まれていた場合は: 端末が次にオンライン復帰した時点で適用 (実装時は WebSocket reconnect 時に未配送コマンドを fetch する設計)

確認ダイアログでも 5 台超の一括送信時に「オフライン端末はオンライン復帰時に適用されます」と注記しています。

## なぜ「タスク」(`/tasks`) と分けるのか

両者は似ていますが目的が違います:

| | 配信タスク (Task) | ライブ切替 (Live Control) |
|---|---|---|
| 目的 | プログラム/APK の **永続的な配信** | 一時的な **再生映像の差し替え** |
| 端末ストレージ | アセットを端末にダウンロード | (ダウンロード済を即時再生) |
| 期間 | 端末から削除するまで | 即時 〜 数分/数時間 |
| 期限指定 | なし | あり (自動復帰) |
| UI | 配信進捗・端末別ステータス | スコープ・モード選択 |

つまり、**Task は「端末に届ける」操作**、**Live Control は「届いてるものを再生命令」操作**です。実運用では:

1. 朝の準備: Task で新キャンペーンプログラムを全端末に配信 (アセットDL)
2. 昼: 計画モードに沿って自動再生
3. イベント発生時: Live Control で全店舗に一斉差し替え (30分間)
4. 30分後: 自動的に計画モードへ復帰

という流れで使い分けます。

## 改善余地 (Phase 2)

- 切替プレビュー: シート上で端末スクリーンを見ながら切替できる
- スケジュール切替: "明日 10:00 に全店舗を X に切替" (既存の plan-schedules を簡易呼出)
- お気に入りプログラム: 頻用プログラムを 1 クリック切替に
- 連動再生グループのタイムシンク監視: ドリフト > 100ms で警告
- 監査ログとの統合: `Live Control` 操作も `/audit-logs` に記録
